# Shipping real payments over HTTP 402

HTTP 402 "Payment Required" has been sitting in the spec since 1997, reserved for a future that never arrived. It seems the future finally showed up: the x402 protocol is settling tens of millions of stablecoin payments a month over that status code, Stripe and AWS have wired it into their infrastructure, and the spec now lives under the Linux Foundation. I've spent the past few weeks running x402 sellers and clients against every facilitator I could get access to (the demos are linked at the bottom), so here's what the protocol actually is in mid-2026, and why I think its most useful property has little to do with the AI agents it was built for.

## The whole protocol in one flow

Strip away the announcements and x402 is a small idea: HTTP already has a status code that means "pay me," so use it.

A client calls your endpoint with nothing attached. You respond 402 with a `PAYMENT-REQUIRED` header containing base64-encoded JSON that says what you'll accept. Decoded, it looks like this:

```json
{
  "x402Version": 2,
  "resource": {
    "url": "https://api.example.com/premium-data",
    "description": "Access to premium market data"
  },
  "accepts": [
    {
      "scheme": "exact",
      "network": "eip155:8453",
      "amount": "5000000",
      "asset": "0x833589fCD6eDb6E08f4c7C32D4f71b54bdA02913",
      "payTo": "0xYourWallet...",
      "maxTimeoutSeconds": 60,
      "extra": { "name": "USDC", "version": "2" }
    }
  ]
}
```

The client picks an option, signs a payment authorization off-chain (for USDC on EVM chains that's ERC-3009 `transferWithAuthorization`: an EIP-712 signature over from, to, value, and a random nonce), then retries the same request with the signed payload base64-encoded in a `PAYMENT-SIGNATURE` header. If it settles, the response comes back 200 with a `PAYMENT-RESPONSE` header carrying the on-chain transaction hash. Two HTTP requests, one signature, and the payer never touches gas.

A few things in that JSON are worth pausing on. `network` is a CAIP-2 identifier (`eip155:8453` is Base mainnet), which means x402 has no chain registry of its own; Ethereum L2s, Solana, and anything else are addressed the same way. `amount` is atomic token units, so `5000000` is $5.00 of six-decimal USDC. And `"exact"` is only the simplest scheme: the spec also defines `upto` for metered usage and batch settlement, and the same message types run over MCP and Google's A2A, not just HTTP. Everything above is the v2 wire format, current since December 2025; you'll still meet older servers that put the requirements in the response body with an `X-PAYMENT` request header, and the reference SDKs quietly speak both.

## The facilitator does the crypto part

The piece that makes this practical is the *facilitator*. Your server never talks to a blockchain. It hands the signed payload to a facilitator service over plain HTTPS, which verifies the signature, submits the transfer on-chain, and pays the gas itself. That last part is what makes the whole thing feel different from every "connect wallet" flow you've suffered through: because ERC-3009 (and Permit2 for tokens that don't support it) moves funds on the strength of an off-chain signature, the payer can hold nothing but USDC. No ETH for gas, ever.

Coinbase's CDP facilitator covers Base, Polygon, Arbitrum, World, and Solana, with 1,000 free settlements a month and $0.001 each after that since January. PayAI runs a competing one across some twenty networks with a bigger free tier and no API keys, thirdweb's rides on its existing chain infrastructure, and x402.rs is an open-source Rust implementation for keeping the whole loop in-house. They're not interchangeable in the details: settlement receipts, error quality, and even which wire format they speak all differ. I ran a seller against each of them, self-hosted included, and wrote up what broke in [a companion post](../2026-07-05-Kicking-the-Tires-on-x402-Facilitators/x402-facilitators.md).

## Nobody's side project anymore

x402 started inside Coinbase in May 2025. In April 2026 it moved into an x402 Foundation under the Linux Foundation, and the charter member list reads like a payments industry summit: Amazon, American Express, Circle, Cloudflare, Fiserv, Google, Mastercard, Microsoft, Shopify, Stripe, Visa, the Solana Foundation. By then the protocol had settled around 165 million payments and roughly $50M in cumulative volume, and the x402.org dashboard currently shows about 75 million transactions in the trailing 30 days, with Solana carrying roughly two-thirds of the traffic.

It's landing in real infrastructure too. Stripe has been settling x402 USDC payments on Base since February. In June, AWS wired it into CloudFront so publishers can charge AI crawlers per request. Cloudflare is onboarding people to a monetization gateway that does the same for pages, datasets, and MCP tools, and Google shipped an official x402 extension for its Agent Payments Protocol.

Notice the shape of all those announcements: agents. The pitch, everywhere, is software that can't fill out a checkout form buying data, API calls, and compute for itself. Machine-to-machine commerce is the future the protocol is being built for.

## Built for agents, useful now

Here's the part I find underrated. You don't need a single agent in sight for x402 to earn its keep, because the same mechanics that let a bot pay for an API call fix the worst part of taking stablecoin payments from people.

Anyone who has accepted crypto from ordinary users knows the ritual: copy this deposit address, connect your wallet to our dapp, make sure you have gas on the right chain first. Every step loses customers. Under x402 all of that collapses to "sign this message." The payer signs one authorization, the facilitator pays the gas, and it works the same whether the wallet holds only USDC on Base or only USDC on Solana. For a checkout, a topup, a paywall, that's the cleanest gasless payment UX I've seen, and it's the reason I'd reach for the protocol today even for entirely human traffic.

And there's an option attached. An x402 endpoint has no idea whether the signature it received came from a person's wallet or an agent's budget; the wire format is identical. Build the payment rail for the gasless experience now, and if agent-driven spending becomes real consumer traffic, everything you shipped is already something an agent can pay, with zero additional work.

The status code waited almost thirty years. It seems to be finding its job.
