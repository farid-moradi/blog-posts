# Shipping real payments over HTTP 402

HTTP 402 "Payment Required" has been sitting in the spec since 1997, reserved for a future that never arrived. Last quarter I finally got to use it for its intended purpose: we shipped x402 payments in the vPay backend, and card topups now literally ride on a 402 status code. This post is a short write-up of how the protocol looks in mid-2026 and what actually bit us in production.

## Where x402 stands now

The protocol now has two wire formats. The original (v1) puts payment requirements in a JSON body on the 402 response and takes the signed payment back in an `X-PAYMENT` header. Version 2, launched in December 2025, moved everything into headers (`PAYMENT-REQUIRED`, `PAYMENT-SIGNATURE`, `PAYMENT-RESPONSE`) and standardized network identifiers on CAIP-2. Both are live; the reference SDKs and Coinbase's facilitator support v1 and v2 side by side, so v1 servers like ours keep working. And the project is no longer just a Coinbase experiment. Coinbase and Cloudflare announced the x402 Foundation in late 2025, it was formalized under the Linux Foundation in April 2026 with launch members including Visa, Mastercard, Stripe, Google, AWS, Circle, Shopify, and the Solana Foundation, and by that same month Coinbase was reporting around 165 million transactions and $50M in cumulative volume. Past the toy stage.

We built on the v1 format. The flow works like this: a client calls your endpoint with no payment attached, and you respond with 402 and a JSON body listing what you'll accept:

```json
{
  "x402Version": 1,
  "accepts": [
    {
      "scheme": "exact",
      "network": "eip155:8453",
      "maxAmountRequired": "5500000",
      "asset": "0x833589fCD6eDb6E08f4c7C32D4f71b54bdA02913",
      "payTo": "0xYourWallet...",
      "resource": "vpay://topup/12345",
      "description": "VPay card topup: $5.00"
    }
  ]
}
```

The client picks an option, signs a payment authorization off-chain (ERC-3009 `transferWithAuthorization` for USDC on EVM chains), and retries the same request with the signed payload base64-encoded in an `X-PAYMENT` header. Your server hands that payload to a *facilitator*, in our case Coinbase's CDP facilitator via their official Go SDK, which verifies the signature and settles the transfer on-chain. The payer never needs gas, and the whole exchange is two HTTP requests.

One admitted impurity: strict v1 identifies networks with slugs like `"base"`, but we adopted v2's CAIP-2 IDs (`eip155:8453` for Base, `eip155:137` for Polygon, `solana:5eykt4...` for Solana mainnet) even on the v1 format, because that's how the CDP facilitator addresses networks and it saved us a translation layer. It's worked without friction, and it turned out to be a useful choice: our chain registry is keyed on those IDs, and adding a new chain to our payment surface is now one `Register()` call and an env var for the receiving wallet. One more deviation worth confessing: the v1 transport says to return the settlement receipt in an `X-PAYMENT-RESPONSE` header, and we return it in our response body instead. That's fine when you control both ends like we do, but if third-party x402 clients will call your API, set the header.

## What actually went wrong

The protocol itself was the easy part. Here's what wasn't.

**Verify can lie.** The facilitator exposes two calls: `Verify` (free, simulates the transfer) and `Settle` (submits it on-chain). Sounds like you should hard-fail on a failed verify. Except CDP's verify uses a staticcall simulation that reverts for EIP-7702 delegated wallets, accounts that settle perfectly fine. We hit this with real users almost immediately. Our fix was to demote verify to a logged warning and treat settle as the authoritative check; if the authorization is bad, settle reverts on-chain anyway. Not elegant, but honest about which call actually decides anything.

**Clients are looser than the spec.** The x402 TypeScript client sometimes omits `asset` and `network` from the payment payload, so we backfill them server-side from the requirement we issued. Budget for this kind of thing anywhere the spec is younger than your users.

**The hard part starts after settlement.** Settlement is irreversible: once the facilitator submits the transfer, you have the user's money whether or not the rest of your request handler succeeds. What happens if the payment session insert fails right after? We ended up writing very deliberate failure-path code: every DB write after settlement logs the tx hash, payer, amount, and session ID at CRITICAL level so a human can reconcile manually. If you take one thing from this post: the 402 handshake is a weekend of work, and the bookkeeping around an irreversible payment inside a fallible request handler is where the real engineering goes.

**Solana broke our assumptions twice.** Our transaction verifier assumed hex tx hashes; Solana hands you base58 signatures. And gasless Solana payments need a `feePayer` in the requirement, which we now fetch from the facilitator's supported-networks endpoint rather than hardcoding.

**Set a floor.** On-chain settlement makes tiny payments technically possible, but a $0.50 topup still triggers the same downstream fiat plumbing as a $500 one. We put a $3 minimum on topups after watching the economics.

## Picking a facilitator

The facilitator is the one piece you outsource, so it's worth knowing the field. It's grown well past "just use Coinbase's." Here's the state of things as of mid-2026; check current docs before committing, this stuff moves:

| Facilitator | Chains | Tokens | Pricing |
|---|---|---|---|
| Coinbase CDP | Base, Polygon, Arbitrum, World, Solana (+ testnets) | USDC/EURC via EIP-3009, any ERC-20 via Permit2 | 1,000 free settlements/mo, then $0.001 |
| PayAI | ~20 networks: Solana, Base, Polygon, Avalanche, Arbitrum, Sei, IoTeX, X Layer, SKALE, peaq… | Stablecoins or your own token | 10,000 free/mo, then $0.001; covers network fees; no API keys |
| thirdweb | 170+ EVM chains, plus Solana | Any ERC-20 with EIP-2612 permit or EIP-3009 | Needs a thirdweb project/secret key |
| x402.rs (self-hosted) | Whatever RPCs you configure (public instance: Base, Polygon, Avalanche, Solana, XDC) | USDC/USDT focus | Free, open-source Rust; you run it |

A few notes on that table. Cloudflare, despite being co-founder of the x402 Foundation, doesn't run a public facilitator; its Agents SDK and Pay Per Crawl examples point at the Coinbase-operated one at x402.org. And "USDC only" stopped being true a while ago: with Permit2 and permit-based approvals, CDP and thirdweb will settle basically any ERC-20, with the facilitator sponsoring the approval so the payer still never needs gas.

We're staying on CDP for now, and not because it's the biggest name. We run a card platform, so the built-in KYT/OFAC screening on settlements is something we'd otherwise have to build; and the official Go SDK meant the integration was mostly wiring. If I were selling API access rather than moving money into a regulated fiat rail, I'd look hard at PayAI first (ten times the free tier, no API keys, and they eat the gas), or self-host x402.rs if I didn't want a third party in the loop at all.

## Is it worth it?

For our use case, which is letting users top up a card from any wallet on Base, Polygon, or Solana with no gas and no dapp connect flow: yes. The user experience collapses to "sign this message" and the server-side surface is one header check and two facilitator calls. Whether x402 becomes the standard rail for agent-to-agent payments the way its backers hope, I don't know. But with the spec now stewarded by a Linux Foundation project and card networks at the table, it's no longer a bet on a single company's side project. The status code waited almost thirty years; it seems to be finding its job.
